# 让 AI 操作企业后台？我试过了，这是它真正该干的四件事

> 一句话结论：AI 在企业 Web 自动化框架里不应该默认拥有执行权。稳定路径交给 Flow 和 Adapter，AI 更适合做意图解析、异常诊断、开发辅助和 Skill 入口。

配图建议：`articles-publish/diagrams/06-ai-role-boundary.html`

前面几篇讲了框架的主线：

- Flow Engine 负责流程编排。
- Adapter 负责系统差异。
- Playwright Runtime 负责确定性执行。
- Report / Trace 负责复盘。

这一篇讲 AI。

很多人一提 AI + 浏览器自动化，就会想到：

```text
AI 看截图 -> AI 决定下一步 -> AI 点击页面 -> 直到任务完成
```

这个方向可以做探索型任务，但我不建议把它作为企业后台自动化的默认架构。

尤其是 SAP、SRM、OA、CRM 这种系统，里面有发布、过账、审批、删除、推送、结算等不可逆操作。AI 可以辅助判断，但不应该默认成为主执行器。

## AI 在框架里的正确位置

我更推荐把 AI 放在四个位置：

| 位置 | AI 做什么 | 是否直接操作浏览器 |
|---|---|---|
| 意图解析 | 把自然语言转成 Flow + params | 否 |
| 异常诊断 | 看截图、a11y tree、错误日志，判断失败原因 | 否 |
| 开发辅助 | 根据录屏和描述生成 Flow / Adapter 草稿 | 否 |
| 交互追问 | 参数缺失时向用户补问 | 否 |

也就是说：

```text
AI 负责理解和建议；
Flow / Adapter / Runtime 负责执行。
```

## 为什么不让 AI 每一步都点

全自主 Agent 的问题不是不能做，而是不适合作为企业后台自动化的默认路线。

| 问题 | 后果 |
|---|---|
| 每步都要模型判断 | 成本高，速度慢 |
| 输出存在概率性 | 同一页面可能给出不同动作 |
| 不可逆操作多 | 点错按钮就是业务事故 |
| 长流程容易疲劳 | 多轮后容易跳步、漏验证 |
| 审计困难 | 很难复盘每一步依据 |

企业自动化最需要的是：

- 可预测。
- 可审计。
- 可复盘。
- 可回放。
- 可逐步扩展。

这些能力主要来自 Flow、Adapter、Report 和 Trace，不来自“让 AI 更努力”。

## 意图解析：自然语言到 Flow

用户可能会说：

```text
帮我查 PO 4500201748 的历史
```

AI 应该做的是把它转成结构化调用：

```json
{
  "flow": "query-po-history",
  "params": {
    "po_number": "4500201748"
  }
}
```

如果参数不完整：

```text
帮我做收货
```

AI 不应该猜 PO 号，而应该追问：

```text
需要采购订单号 po_number，才能执行 goods-receipt Flow。
```

这就是 Skill 入口的价值：它让用户用自然语言触发 Flow，但最终执行的仍然是结构化流程。

## 异常诊断：截图 + a11y tree + 错误上下文

当 Flow 步骤失败时，AI 可以做诊断。

输入不应该只有一句报错，而应该包含：

```text
task: 当前业务任务
flow: 当前 Flow 名称
step: 当前步骤 id
expected: 当前步骤完成后的业务证据
error: Playwright 错误信息
screenshot: 当前页面截图
a11y_tree: 过滤后的交互元素列表
history: 最近 3-5 步执行记录
```

AI 输出也不应该自由发挥，而应该受限：

```json
{
  "action": "retry|retry_with_wait|switch_state|abort|manual",
  "reason": "失败原因",
  "evidence": "来自截图/a11y tree/错误日志的证据",
  "suggestion": "具体修复建议"
}
```

这样诊断结果才有工程价值。

## 为什么要 a11y tree

截图让 AI 看见页面状态，a11y tree 让 AI 拿到可操作结构。

截图能告诉 AI：

- 当前停在哪个页面。
- 是否有弹窗。
- 是否出现错误消息。
- 页面视觉上是否符合预期。

a11y tree 能告诉 AI：

- 哪些元素是按钮。
- 哪些元素是输入框。
- 元素的 name / role / state。
- 当前页面有哪些可交互对象。

过滤后的结构可以类似：

```json
[
  { "role": "textbox", "name": "采购凭证", "enabled": true },
  { "role": "button", "name": "其他采购订单", "enabled": true },
  { "role": "button", "name": "执行", "enabled": true },
  { "role": "status", "name": "请输入采购凭证" }
]
```

这比全量 DOM 更适合模型理解，也比单纯截图更容易落到证据。

## AI 诊断不是最终修复

AI 诊断的结果，应该回流到工程系统。

例如：

第一次，Flow 在 ME23N 输入 PO 时失败。

AI 看到截图和 a11y tree 后判断：

```json
{
  "action": "switch_state",
  "reason": "页面已加载上次查看的采购订单，当前字段不可直接编辑",
  "evidence": "截图中存在订单详情，a11y tree 中有“其他采购订单”按钮",
  "suggestion": "点击“其他采购订单”后重新输入目标 PO"
}
```

这不是结束。

真正的修复是把它沉淀进 SAP Adapter：

```text
ME23NPage.resetToInputMode()
ME23NPage.openPurchaseOrder(poNumber)
```

下一次再遇到同类问题，就不需要 AI 诊断了。

这才是半自主框架的循环：

```text
AI 发现未知异常
  -> 人类确认
  -> 写入 Adapter / Flow
  -> 下一次变成确定性能力
```

## 开发辅助：从录屏到 Flow / Adapter 草稿

AI 还有一个很实用的位置：辅助开发者写新 Flow。

新增一个系统流程时，可以先人工操作一遍，保存：

- 操作录屏。
- 关键截图。
- 字段说明。
- 预期结果。
- 失败样例。

然后让 AI 生成草稿：

- Flow YAML。
- Adapter 方法名。
- Page Object 骨架。
- 需要人工确认的风险点。
- 可能的测试用例。

但这里仍然有边界：

```text
AI 生成草稿；
开发者验证 selector、业务含义和不可逆操作；
最终进入代码和 Flow 的，必须经过人类审核。
```

## Skill 集成状态

这个项目的最终形态是接入 OpenClaw / 飞书，让用户用自然语言触发 Flow：

```text
用户：帮我查 PO 4500201748 的历史
AI：识别意图 = query-po-history
AI：提取参数 = { po_number: "4500201748" }
Skill：调用 sap-auto run-flow query-po-history
系统：返回报告和截图
```

当前状态要诚实区分：

| 模块 | 状态 | 说明 |
|---|---|---|
| CLI 执行 Flow | 已实现 | 可用命令行运行 |
| YAML Flow + SAP Adapter | 已实现 | 当前第一个实践样例 |
| `screenshot_and_report` | 已实现 | 失败时截图写报告 |
| OpenClaw Skill | 规划中 | SKILL.md 已有设计方向，待部署 |
| 飞书自然语言触发 | 规划中 | 依赖平台接入 |
| `ai_diagnose` | 规划中 | 真实模型调用未稳定接入 |

这个表建议发布时保留。它会降低夸大风险，反而提升可信度。

## AI 介入边界

建议把 AI 权限写成规则：

```text
AI 可以：
[x] 解析用户意图
[x] 提取 Flow 参数
[x] 发现参数缺失并追问
[x] 分析失败截图和错误日志
[x] 给出 retry / abort / manual 建议
[x] 辅助生成 Flow / Adapter 草稿

AI 不可以默认：
[ ] 自由点击不可逆按钮
[ ] 绕过 Flow 的确认点
[ ] 在没有证据时声称完成
[ ] 修改业务数据后不报告
[ ] 跳过 trace / report
```

## 本篇可带走

如果你正在给 Web 自动化框架加 AI，可以先用这个架构：

```text
正常路径：
  User -> Skill -> Flow Engine -> Adapter -> Playwright -> Report

异常路径：
  Step failed -> Screenshot + a11y tree + Error -> AI Diagnose
              -> retry / abort / manual / write back to Adapter

开发路径：
  Human recording -> AI draft -> Developer review -> Flow / Adapter
```

## FAQ

**Q：AI 在这个框架里是不是噱头？**

不是，但它也不是主执行器。AI 的价值在自然语言入口、异常诊断和开发辅助；稳定执行仍然靠 Flow、Adapter 和 Playwright。

**Q：截图 + a11y tree 能让 AI 操作任何网页吗？**

不能。它能提高页面理解能力，但不等于稳定执行。稳定执行仍然需要定位、等待、验证、错误处理和审计链路。

**Q：为什么 AI 诊断还标注为规划中？**

因为当前已实现的是截图和报告链路，真实 Claude API 接入还没有作为稳定模块落地。发布文章时把状态说清楚，可信度比强行包装更高。

**Q：这个框架下一步最值得补什么？**

三类材料最有价值：脱敏 Demo 录屏、真实 HTML report 截图、一次失败 trace 的复盘。它们比继续堆概念更能证明框架有效。

## 总结

AI 不应该成为企业 Web 自动化框架里的“自由点击者”。

更稳的分工是：

```text
Flow Engine：决定流程怎么走
Adapter：处理系统怎么点
Playwright Runtime：保证执行可追踪
Report / Trace：保证失败可复盘
AI：理解意图、解释异常、辅助开发
```

这也是为什么我把它称为半自主框架：不是不用 AI，而是不把确定性和责任边界交给 AI。
