# 如何设计一个 96 分的 AI Skill：通用 Web 自动化能力构建

> 一个 AI Skill 从 72 分到 96 分的四轮迭代全记录——设计方法论 + 评审标准 + 避坑指南。
> 归档状态：早期旧草稿，保留 Skill 设计方法论素材。当前项目内可执行边界以 `skills/sap-ui-auto/SKILL.md` v3.10 和后续 eval 为准。

## 前言

AI Agent 的能力上限，由它的 Skill（能力描述文档）决定。

Skill 不是"写给人看的帮助文档"，而是**写给 AI 看的操作手册**。它决定了 AI 能做什么、怎么做、什么时候做、出了问题怎么办。

本篇完整记录了我们如何将一个 Web 自动化 Skill 从 72 分迭代到 96 分的全过程。每一轮的评审标准、扣分原因、优化方法——适用于任何 AI Skill 的设计。

## 一、什么是 AI Skill

### 本质

Skill = 一份 Markdown 文档，安装到 AI Agent 上，赋予它新能力。

```markdown
---
name: web-ui-auto
description: 当用户要求自动化操作 Web 页面时触发
tools: [bash]
---

# Web UI 自动化 Skill
你是 Web UI 自动化助手，通过 Playwright 控制浏览器...
```

### 与 MCP / Plugin / API 的区别

| 对比维度 | Skill | MCP Tool | Plugin | API |
|----------|-------|----------|--------|-----|
| 格式 | Markdown | JSON Schema | 代码模块 | RESTful |
| AI 理解方式 | 阅读文档自主执行 | 按参数调用 | 按接口调用 | HTTP 请求 |
| 灵活度 | 高（AI 自由发挥） | 中（参数固定） | 低 | 低 |
| 部署成本 | 零（.md 文件） | 需运行 Server | 需安装 | 需运维 |
| 适合场景 | 复杂/多变/需判断 | 标准化重复操作 | 平台集成 | 系统对接 |

### 关键洞察

> **Skill 不是代码，是知识。** 它把"如何做某件事"的知识传递给 AI，AI 结合当前上下文灵活运用。

这意味着：
- 写 Skill 不需要编程能力
- Skill 修改即时生效，无需编译部署
- 同一个 Skill 可以装到不同的 AI Agent 上

## 二、评审维度体系

### 9 维评审模型

| 维度 | 含义 | 评判标准 |
|------|------|----------|
| 能力覆盖 | 目标场景是否都能处理 | 无明显盲区 |
| 触发精度 | 该触发时触发，不该时不触发 | 低误触发 + 低漏触发 |
| 执行指导性 | AI 读完能否直接执行 | 零歧义 |
| 异常处理 | 错误恢复是否充分 | 闭环恢复链 |
| 信噪比 | 有用信息占比 | 无冗余 |
| 可维护性 | 新增能力是否要大改 | 扩展方便 |
| 结构设计 | 信息组织 | 分层清晰 |
| AI可执行性 | 命令/路径/参数是否明确 | 可直接复制执行 |
| Token效率 | 加载消耗 | <4000 token |

### 评分对照

| 分数 | 级别 | 含义 |
|------|------|------|
| 95+ | A+ | 生产级，几乎无需改进 |
| 90-94 | A | 优秀，小瑕疵不影响使用 |
| 85-89 | B+ | 良好，有改进空间 |
| 75-84 | B | 可用，存在明显问题 |
| <75 | C | 需要重大修改 |

## 三、四轮迭代全记录

### 第一轮：72 分 — 能用但局限

**问题诊断：**

| 扣分项 | 原因 |
|--------|------|
| 只支持一个系统 | 标题就是"SAP Web GUI 自动化" |
| Flow 列表硬编码 | 16 个 Flow 写死在文件里 |
| 异常处理太浅 | 只列了 5 种情况，无恢复流程 |
| 无能力边界 | AI 不知道什么做不到 |
| 触发词太泛 | "提到 SAP"就触发 |

**核心问题**：把框架局限成了单系统工具。

---

### 第二轮：86 分 — 通用化重构（+14）

**关键改动**：

```diff
- # SAP Web GUI 自动化 Skill
+ # Web UI 自动化 Skill
+ 你是 Web UI 自动化助手，通过 Playwright 控制浏览器操作任意 Web 页面。

- 已有 YAML 流程：
-   - create-invoice — MIRO 创建发票
-   - verify-invoice — MIR4 校验发票
-   - ... (16行)
+ 已有 YAML 流程目录：`/flows/`
+ 执行前先查看可用流程：
+ ```bash
+ ls flows/*.yaml
+ ```

+ ## 错误恢复策略
+ | 情况 | 恢复方式 |
+ | 元素定位失败 | 截图→AI分析→换策略 |
+ | 同一操作失败2次 | 截图诊断 |
+ | 页面白屏 | reload→重新导航 |
+ | ... |

+ ## 不支持的场景
+ - Canvas / WebGL
+ - 原生桌面应用
+ - 验证码识别
```

**得分提升来源**：

| 改动 | 提升维度 | 幅度 |
|------|----------|------|
| 通用化定位 | 能力覆盖 | +1 |
| Flow 动态化 | 可维护性 | +3 |
| 错误恢复 8 场景 | 异常处理 | +3 |
| 触发词收窄 | 触发精度 | +2 |
| 删冗余代码示例 | 信噪比 | +2 |
| 加能力边界 | 执行指导性 | +1 |

**关键洞察**：

> **不要教 AI 它已经会的东西。**
>
> Playwright 的 iframe API、多标签页处理、文件上传——AI 全都懂。
> Skill 里只需要写 AI 不知道的：特定系统的怪癖、项目的路径配置、团队的约定。

---

### 第三轮：91 分 — 补全细节（+5）

**问题**：AI 执行时会卡在"不知道写到哪个文件"、"不知道截图存在哪"这类细节上。

**改动**：

| 补充 | 解决什么 |
|------|----------|
| 脚本写入路径 `src/tasks/` | AI 知道文件放哪 |
| 截图保存路径 `screenshots/` | 结果可追溯 |
| 通用登录三种方式 | 不只 SAP 能登录 |
| SAP 意图"条件激活" | 非 SAP 场景不展示 |
| 等待策略去重 | 信噪比提升 |

**洞察**：

> **Skill 的每个细节都应该让 AI 执行时零歧义。**
>
> "写一个脚本运行"→ AI 会问：写到哪？叫什么名？用什么命令跑？
> "写到 `src/tasks/xxx.ts`，用 `npx tsx src/tasks/xxx.ts` 运行"→ AI 直接执行。

---

### 第四轮：96 分 — 生产级（+5）

**改动**：

| 新增 | 价值 |
|------|------|
| 运行环境说明（npx tsx） | 防踩 ESM 坑 |
| 最小脚本模板（3行） | 零门槛开始写 |
| CDP 连接已有浏览器 | 覆盖"不想重新登录"场景 |
| 浏览器生命周期策略 | 执行完不意外关浏览器 |
| Token 与性能提示 | 用户预期管理 |
| 沉淀判断标准 | 指导脚本→Flow 升级时机 |
| 多环境支持 | 企业必备（测试/生产） |
| 示例加输出 | 用户知道 AI 会返回什么 |
| Changelog | 版本追踪 |

## 四、7 条设计原则

从四轮迭代中总结出的核心原则：

### 原则 1：只写 AI 不知道的

```
❌ "Playwright 的 page.click() 方法用于点击元素"（AI 知道）
❌ "iframe 可以用 frameLocator() 切换"（AI 知道）

✓ "SAP 字段是 readonly 的，需要先 click() 激活再 pressSequentially()"（AI 不知道）
✓ "脚本写到 src/tasks/ 目录下"（项目配置，AI 不知道）
```

### 原则 2：零歧义可执行

```
❌ "在项目目录下运行"
✓ "cd E:/sap-playwright-agent && npx tsx src/tasks/{name}.ts"

❌ "保存截图"
✓ "截图保存至 E:/sap-playwright-agent/screenshots/（按时间戳命名）"
```

### 原则 3：动态优于静态

```
❌ 硬编码 16 个 Flow 的名字和描述
✓ "运行 ls flows/*.yaml 查看可用流程"

❌ "当前支持的系统：SAP ECC、SAP SRM"
✓ "通过 domains 字段标记，运行时按需加载"
```

### 原则 4：声明能力边界

AI 倾向于乐观承诺"我可以做到"。明确写出不支持什么：

```markdown
## 不支持的场景
- Canvas / WebGL 绘图内容
- 原生桌面应用
- 图形验证码识别
- 硬件设备操作（指纹/USB Key）
```

### 原则 5：分层设计

```
通用层（任何页面适用）
  ↓
领域层（特定系统知识，按需激活）
  ↓
流程层（具体 Flow 配置）
```

新增系统 = 加一段领域知识，不改通用层。

### 原则 6：控制 Token

| 策略 | 效果 |
|------|------|
| 总长度 <250 行 | 加载 <4000 token |
| 不教 AI 已知 API | 省 30% |
| 条件激活领域知识 | 非相关场景不消耗 |
| 动态获取替代静态列表 | Skill 不随数据增长 |

### 原则 7：持续迭代

每次 AI 执行出错或犹豫，就是 Skill 需要补充的信号：

| AI 的困惑 | Skill 该补什么 |
|-----------|---------------|
| 不知道文件写到哪 | 补路径说明 |
| 不确定浏览器关不关 | 补生命周期策略 |
| 在普通页面用了 SAP 技巧 | 加条件激活 |
| 脚本跑不起来 | 补运行环境说明 |

## 五、反模式速查

### 反模式 1：教科书式

```
❌ "Playwright 是由 Microsoft 开发的浏览器自动化工具..."
```

AI 不需要科普，需要"怎么用于这个项目"。

### 反模式 2：代码堆砌

```
❌ 每种操作给完整代码示例（50+ 行代码块）
✓ 一句话概括 + 只在 AI 不知道的场景给代码
```

### 反模式 3：面面俱到

```
❌ 列出所有可能用到的 Playwright API
✓ 只写特殊的、反直觉的、项目特有的
```

### 反模式 4：无边界

```
❌ "本 Skill 可以处理各种 Web 自动化需求"（AI 会接受验证码识别任务）
✓ 明确写出不能做的事情
```

### 反模式 5：触发词贪心

```
❌ description: "网页、浏览器、页面、点击、自动化..."（日常对话也会触发）
✓ description: "当用户要求自动化操作 Web 页面时触发。仅咨询问题不触发。"
```

## 六、Skill 结构模板

基于四轮迭代验证的最佳结构：

```markdown
---
name: {skill名}
version: "x.y"
description: {精确的触发条件描述}
tools: [bash]
domains: [{领域标记}]
changelog: {版本变更记录}
---

# {Skill 名称}

{一句话定位}

## 核心能力
{能力列表，9-10 项}

## 执行模式
### 模式一：预定义流程
{如何查看 + 如何执行}

### 模式二：即时操作
{脚本路径 + 模板 + 运行命令 + 沉淀标准}

## 工作流程
### 1. 意图识别
{意图表}

### 2. 执行
{关键经验 + 等待策略}

### 3. 结果反馈
{输出格式 + 截图路径}

### 4. 生命周期管理
{资源管理策略}

### 5. 性能提示
{Token/成本/速度}

## 错误恢复策略
{异常表 + 通用恢复流程}

## 自主操作模式
{多步任务的编排方式}

## 领域增强：{系统名}（按需激活）
{特定系统的专有知识}

## 不支持的场景
{明确的能力边界}

## 意图不明确时
{兜底策略}

## 示例
{输入 → 执行 → 输出 的完整闭环}
```

## 七、评审 Checklist

设计完 Skill 后的自检清单：

**必须通过（P0）：**
- [ ] AI 读完能直接执行，不需要再问"文件在哪？"
- [ ] 没有教 AI 它已经知道的东西
- [ ] 触发条件精确，有"不触发"的排除条件
- [ ] 明确声明了不支持的场景
- [ ] 有完整的错误恢复策略

**应该通过（P1）：**
- [ ] 动态获取替代硬编码
- [ ] 有版本号和 changelog
- [ ] 总 Token <4000
- [ ] 示例包含输出
- [ ] 新增能力不需要大改结构

**加分项（P2）：**
- [ ] 有性能/成本提示
- [ ] 有沉淀标准（何时从一次性操作升级为可复用流程）
- [ ] 有多环境支持
- [ ] 领域知识条件激活

## 九、总结

### 迭代得分

```
72 → 86 → 91 → 96
```

### 每轮核心改进

| 轮次 | 核心改进 | 方法论 |
|------|----------|--------|
| 1→2 | 通用化 + 动态化 | "不绑定系统"、"不硬编码" |
| 2→3 | 补全细节 | "AI 执行时的每个疑问都是改进点" |
| 3→4 | 生产级完善 | "从第一次使用到长期使用的全生命周期" |

### 核心认知

1. **Skill 是写给 AI 的** — 零歧义比人类可读性更重要
2. **少即是多** — 删冗余比加内容更能提分
3. **动态优于静态** — 命令获取 > 硬编码列表
4. **边界比能力更重要** — 防过度承诺
5. **AI 的每次犹豫都是信号** — 它不确定时，Skill 就该补充

这套方法论适用于设计任何 AI Skill——不只是 Web 自动化，任何需要让 AI 执行操作的场景都适用。
