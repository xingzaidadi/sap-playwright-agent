## 一、从一个页面到多个系统

### 通用框架的扩展路径

```
第一步：在一个系统上验证可行性
第二步：抽象通用层，系统特异行为下沉到领域层
第三步：新系统接入只需添加领域知识（甚至不加）
```

实际案例：

| 阶段 | 系统 | 需要额外代码 | 需要领域知识 |
|------|------|-------------|-------------|
| 验证期 | SAP ECC | 是（第一个） | 是 |
| 扩展期 | SAP SRM | 否（复用框架） | 是（新标签页、日期弹窗） |
| 成熟期 | OA 系统 | 否 | 否（通用策略就够） |
| 成熟期 | 电商后台 | 否 | 可选（加了更快） |

### 新系统接入流程

```
1. AI 尝试直接操作（通用策略）
   ↓ 成功 → 结束
   ↓ 失败 →

2. 分析失败原因
   ↓ 通用问题（弹窗/加载/定位）→ 错误恢复自动处理
   ↓ 系统特有行为 →

3. 记录领域知识（几行 Markdown）
   "这个系统的输入框需要先 double-click 激活"
   ↓

4. 加入 Skill 的领域增强章节
```

**大部分系统不需要走到第 3 步。**

## 二、领域知识：何时加、怎么加

### 判断标准

| 信号 | 是否需要领域知识 |
|------|-----------------|
| AI 一次就成功了 | 不需要 |
| AI 重试一次后成功 | 不需要（恢复策略够用） |
| AI 同一个地方连续失败 3 次 | 需要 |
| 某个操作人知道但 AI 反复错 | 需要 |
| 系统有非标准的交互模式 | 需要 |

### 领域知识的格式

```markdown
## 领域增强：{系统名}

**{系统名} 专有经验：**
- {特殊行为1}
- {特殊行为2}
- {特殊行为3}
```

### 实际案例

**SAP ECC**（6 条领域知识）：
```markdown
- 字段 readonly，需 click() 激活 + pressSequentially()
- 工具栏按钮需 force: true
- 填写后必须 Enter 触发校验
- 命令框标签是"输入事务代码"
- 事务码前加 /n 当前窗口打开
- 状态栏在页面底部
```

**假设的 OA 系统**（3 条就够）：
```markdown
- 审批意见有 200 字限制
- 附件预览在新标签页打开
- 退回操作必须填写理由
```

**假设的电商后台**（2 条）：
```markdown
- 批量操作最多选 50 条（需分页处理）
- 删除操作有二次确认弹窗
```

### 领域知识的投入产出

| 系统复杂度 | 领域知识量 | 编写时间 | 成功率提升 |
|-----------|-----------|---------|-----------|
| 标准 Web 应用 | 0-2 条 | 0-5 分钟 | +0-5% |
| 有特殊交互的系统 | 3-5 条 | 5-10 分钟 | +10-15% |
| 高度非标系统（如 SAP） | 6-10 条 | 10-20 分钟 | +15-20% |

**关键认知**：领域知识的 ROI 极高——几分钟的文档编写，换来持续的成功率提升。

## 三、预定义 Flow vs 即时脚本

### 什么时候用 Flow

```
Flow = 固定流程 × 高频使用 × 需要参数化
```

| 适合做 Flow | 不适合做 Flow |
|-------------|--------------|
| 每天都要执行的操作 | 探索性的一次性操作 |
| 步骤固定、参数变化 | 每次步骤都不一样 |
| 需要多人/多环境复用 | 个人临时需求 |
| 有明确的输入输出 | 不确定要做什么 |

### 沉淀判断标准

> **用 ≥2 次，或包含 ≥5 步固定操作 → 提取为 Flow**

### Flow 的 YAML 格式

```yaml
name: oa-daily-approval
description: OA 每日审批处理
system: generic-web
params:
  - name: url
    required: true
  - name: username
    required: true
  - name: password
    required: true
steps:
  - action: goto
    url: "{{url}}"
  - action: fill
    target: 用户名
    value: "{{username}}"
  - action: fill
    target: 密码
    value: "{{password}}"
  - action: click
    target: 登录
  - action: click
    target: 待办事项
  - action: loop
    condition: 有待审批项
    steps:
      - action: click
        target: 第一条待办
      - action: click
        target: 通过
      - action: click
        target: 确认
```

### 即时脚本的使用场景

```typescript
// src/tasks/check-order-status.ts
// 一次性任务：查看某个订单的状态
import { chromium } from 'playwright'

const browser = await chromium.launch({ headless: false })
const page = await browser.newPage()
await page.goto('<ADMIN_PORTAL_URL>')
// ... AI 会根据意图补充具体操作代码
```

## 四、多环境管理

### 企业场景的现实

```
测试环境：随便操作，数据可重建
预发环境：小心操作，接近真实
生产环境：慎之又慎，操作可追溯
```

### 框架的多环境支持

```typescript
// 环境配置通过 .env 或参数传入
const ENV = process.env.TARGET_ENV || 'test'

const config = {
  test: {
    url: '<SAP_TEST_URL>',
    credentials: 'test-credentials',
  },
  prod: {
    url: '<SAP_PROD_URL>',
    credentials: 'prod-credentials',
  }
}
```

### 安全策略

| 环境 | 策略 |
|------|------|
| 测试 | 直接执行，无确认 |
| 预发 | 执行前展示操作计划 |
| 生产 | 每步确认 + 执行后截图留证 |

Skill 中的表述：

```markdown

- env: 'test' — 默认，直接执行

```

## 五、成本控制

### Token 消耗结构

单次 AI 决策的 Token 消耗：

| 组成 | Token |
|------|-------|
| System Prompt | 400 |
| 截图 | 1000-1500 |
| a11y tree | 500-1000 |
| 历史操作 | 200-500 |
| AI 输出 | 50-100 |
| **合计** | **2150-3500** |

### 不同使用模式的月成本

| 使用模式 | 月 Token | 月成本(Sonnet) |
|----------|---------|---------------|
| 轻度（日均 5 次简单操作） | 150K | <$1 |
| 中度（日均 20 次操作） | 700K | ~$3 |
| 重度（日均 100 次操作） | 3.5M | ~$12 |
| 批量（日均 500 次） | 17M | ~$60 |

### 降本策略

**策略 1：预定义 Flow 跳过 AI**

对于已验证的稳定流程，直接按 YAML 步骤执行，不调用 AI 决策：

```
预定义 Flow：0 AI token（纯 Playwright 执行）
即时操作：需要 AI 决策（消耗 token）
```

**策略 2：批量操作优化**

```
❌ 50 个订单 × 每个 5 步 × 每步截图 = 250 次 AI 调用
✓ 第一个订单完整 AI 决策 → 后续 49 个复用同一脚本 = 5 次 AI 调用
```

**策略 3：关键步骤截图**

```
❌ 每一步都截图给 AI 看（每步 1500 token）
✓ 只在关键节点截图（操作前、操作后、异常时）
```

**策略 4：模型降级**

| 场景 | 推荐模型 |
|------|----------|
| 复杂判断（首次操作新页面） | Claude Sonnet |
| 简单重复（相同流程不同参数） | Claude Haiku（便宜 10x） |
| 已验证 Flow | 不需要模型 |

## 六、可靠性保障

### 成功率数据

| 场景 | 首次成功率 | 含重试成功率 |
|------|-----------|-------------|
| 标准 Web 表单 | 90% | 97% |
| 复杂 SPA 应用 | 80% | 92% |
| 有领域知识的系统 | 93% | 98% |
| 预定义 Flow | 99% | 99.5% |

### 失败的主要原因

| 原因 | 占比 | 对策 |
|------|------|------|
| 页面加载慢/超时 | 35% | 增加等待 + 重试 |
| a11y tree 不完整 | 25% | 回退到视觉定位 |
| AI 决策错误 | 20% | 更精确的领域知识 |
| 元素被遮挡 | 15% | 先清理弹窗/遮罩 |
| 其他 | 5% | 报告用户 |

### 提升可靠性的手段

**手段 1：重试 + 递进等待**

```typescript
// 等待策略递进
const waits = [1500, 3000, 5000]
for (const ms of waits) {
  await page.waitForTimeout(ms)
  if (await elementVisible()) break
}
```

**手段 2：截图审计**

```
每步操作都截图保存 → 失败时回放截图序列 → 定位具体失败点
```

**手段 3：健康检查**

```typescript
// 执行前检查环境
async function healthCheck(page: Page) {
  // 页面可访问
  await page.goto(targetUrl, { timeout: 10000 })
  // 已登录
  if (await page.locator('.login-form').isVisible()) {
    await login(page)
  }
  // a11y tree 可用
  const snapshot = await page.accessibility.snapshot()
  if (!snapshot) logger.warn('a11y tree unavailable, falling back to visual')
}
```

**手段 4：降级策略**

```
AI 决策 → 失败 → 重试 AI 决策 → 仍失败
→ 降级到预定义 Flow（如果有）
→ 仍失败 → 报告用户并给出分析
```

## 七、团队协作模式

### 角色分工

| 角色 | 职责 |
|------|------|
| 自动化工程师 | 维护框架 + 编写复杂领域知识 |
| 测试人员 | 配置 Flow（YAML）+ 触发执行 |
| 业务人员 | 自然语言描述需求 + 验收结果 |

### 工作流

```
业务人员："帮我自动处理每天的 OA 审批"
    ↓
测试人员：先用即时模式验证 → 沉淀为 Flow
    ↓
自动化工程师：补充领域知识（如果需要）
    ↓
日常运行：Flow 自动执行，异常时报告
```

### 知识沉淀路径

```
第一次操作：AI 即时决策（消耗 token）
    ↓ 成功 → 记录操作步骤
第二次操作：AI 决策但有历史参考
    ↓ 仍成功 → 满足沉淀条件
沉淀为 Flow：后续零 token 执行
```

## 十一、总结

### 本系列核心观点

1. **通用性来自 AI 的理解能力** — 不绑定选择器、不绑定系统
2. **双通道感知** — 截图看全貌 + a11y tree 精准定位
3. **Skill 是知识不是代码** — 改文档就改行为
4. **领域知识是加速器** — 有则更快，无也能跑
5. **持续迭代** — 72→96 不是一步到位

### 核心价值

| 对谁 | 价值 |
|------|------|
| 测试工程师 | 一套框架覆盖所有系统的 UI 自动化 |
| 企业 IT | 低成本替代 RPA 平台 |
| AI 开发者 | 学习 Skill 设计 + Agent 架构 |
| 业务人员 | 自然语言驱动，零代码使用 |
