# AI 看屏幕就能操作任何网页：一个通用 Web 自动化框架的诞生

> 不写选择器、不录脚本、不绑定系统——给 AI 一句话，它看着屏幕就能替你操作任何 Web 页面。
> 归档状态：早期旧草稿，保留为选题和表达素材。当前 V3 后的正式表述不再使用“任何网页 / 任意系统 / AI 默认自主执行不可逆业务动作”这类说法。

## 前言

Web 自动化领域有个老问题：**每换一个系统就要重新写一套代码**。

测试一个 OA 系统要写一套 Selenium 脚本，自动化 ERP 又要写一套，做电商后台还得再来一套。选择器一变全崩，维护成本随系统数量线性增长。

本系列介绍一种全新方案：**用 AI Agent + Playwright 构建通用 Web 自动化框架**。核心思路——AI 看截图理解页面、读 a11y tree 识别元素、用自然语言接收指令、自主决策操作步骤。

**一套框架，操作任意 Web 页面。**

## 一、Web 自动化的本质困境

### 困境 1：选择器绑定

```javascript
// 今天能跑
await page.locator('#app > div.main > form > input:nth-child(3)').fill('test')

// 明天前端改了结构就崩了
// Error: Element not found
```

选择器把自动化代码和页面结构**强耦合**。结构一变，自动化全废。

### 困境 2：系统绑定

每个系统的操作方式不同：
- OA 系统：标准表单 + 审批流
- ERP（SAP）：iframe 嵌套 + readonly 字段 + 事务码
- 电商后台：SPA 路由 + 动态表格 + 批量操作
- CRM：复杂表单 + 关联数据 + 弹窗交互

传统方案为每个系统单独开发，代码无法复用。

### 困境 3：异常绑定

```python
# 传统方案：预判所有可能的异常
if popup_exists():
    close_popup()
elif loading():
    wait()
elif error_message():
    handle_error()
# 永远写不完...
```

页面上可能出现的意外情况无穷多，硬编码处理永远覆盖不全。

### 根本原因

传统自动化本质是**规则匹配**：预定义选择器 + 预定义流程 + 预定义异常。它做不到"遇到没见过的情况自主判断"。

## 二、解决思路：让 AI 来"看"和"想"

人类操作网页时不需要知道 DOM 结构：
1. **看**一眼页面 → 理解当前状态
2. **想**一下 → 决定下一步做什么
3. **操作** → 点击/填写/等待
4. **看结果** → 判断是否成功

AI 也可以这样做：

```
截图（看）→ Claude 多模态理解（想）→ Playwright 执行（操作）→ 新截图（看结果）
```

**这就是通用性的来源**——不绑定任何系统的 DOM 结构，只依赖视觉理解。

### 核心组件

| 组件 | 作用 | 为什么通用 |
|------|------|-----------|
| 截图 | 获取页面视觉信息 | 任何页面都能截图 |
| a11y tree | 获取可交互元素列表 | 任何页面都有 a11y |
| Claude 多模态 | 理解页面 + 决策 | AI 能看懂任何页面 |
| Playwright | 执行操作 | 能控制任何浏览器页面 |
| Skill | 能力描述 + 领域知识 | 通用层 + 可插拔领域层 |

**每一层都是通用的**，这就是框架通用性的根基。

## 四、与传统方案的根本区别

| 维度 | 传统自动化 | AI Agent 框架 |
|------|-----------|---------------|
| 识别元素 | CSS/XPath 选择器 | AI 视觉 + a11y name |
| 理解页面 | 不理解，只匹配 | AI 理解语义和上下文 |
| 处理异常 | 预定义分支 | AI 实时分析截图决策 |
| 新系统适配 | 重新开发 | 零代码，直接操作 |
| 流程变更 | 改代码/重录 | 改一句自然语言 |
| 通用性 | 单系统 | 任意 Web 页面 |

### 举个例子

传统方式操作 3 个不同系统：

```javascript
// OA 系统
await page.locator('#approval-btn').click()

// ERP 系统（完全不同的写法）
await frame.locator('span.lsButton__text:has-text("过账")').click({ force: true })

// 电商后台（又不同）
await page.getByRole('button', { name: '批量发货' }).click()
```

AI Agent 方式：

```
"点击审批按钮"  → AI 看截图找到并点击
"点击过账按钮"  → AI 看截图找到并点击
"点击批量发货"  → AI 看截图找到并点击
```

AI 不需要知道底层实现差异，它只需要"看到"按钮在哪里。

## 五、Skill 的分层架构

Skill 是框架的"大脑配置"，决定 AI 的行为方式：

```
┌────────────────────────────────────────────┐
│  通用层（任何 Web 页面都适用）               │
│  - 元素定位策略                             │
│  - 等待策略                                 │
│  - 错误恢复                                 │
│  - 截图 + a11y tree 采集                   │
├────────────────────────────────────────────┤
│  领域层（可选，提高特定系统的成功率）         │
│  ┌────────┐ ┌────────┐ ┌────────┐         │
│  │  SAP   │ │   OA   │ │  电商  │ ...     │
│  └────────┘ └────────┘ └────────┘         │
├────────────────────────────────────────────┤
│  流程层（具体可复用的操作流程）              │
│  flows/*.yaml                              │
└────────────────────────────────────────────┘
```

**关键设计**：
- 通用层处理 80% 的场景
- 领域层是**加速器**（提高成功率），不是**必需品**
- 没有领域知识，AI 多试几次也能搞定普通页面

## 六、实际能力演示

### 场景 1：OA 审批（零配置直接操作）

```
用户：登录 oa.company.com，审批通过我的第一条待办

AI 执行过程：
1. [截图] 登录页 → 填用户名密码 → 点登录
2. [截图] 首页 → 找到"待办事项" → 点击
3. [截图] 待办列表 → 点击第一条
4. [截图] 审批详情 → 找到"通过"按钮 → 点击
5. [截图] 确认弹窗 → 点确定
6. [完成] "审批通过成功"

输出："已通过待办'2025年Q2预算申请'。截图已保存。"
```

### 场景 2：电商后台批量操作

```
用户：登录 admin.shop.com，把今天的待发货订单全部标记已发货

AI 执行过程：
1. 登录 → 进入订单管理
2. 筛选"待发货" + 今天日期
3. 全选 → 批量操作 → 标记已发货
4. 检查分页 → 翻页继续处理
5. 汇总结果

输出："共处理 23 条订单，全部标记已发货。✓20条成功 ✗3条失败(库存不足)。"
```

### 场景 3：测试验证

```
用户：帮我测试注册页面 <REGISTER_PAGE_URL>，验证表单校验和注册成功

AI 执行过程：
1. 空表单提交 → 验证错误提示出现 ✓
2. 填无效邮箱 → 验证格式校验 ✓
3. 填正确信息 → 提交 → 验证跳转成功页 ✓
4. 重复注册 → 验证"用户已存在"提示 ✓

输出："4/4 测试场景通过。截图记录已保存至 screenshots/。"
```

### 场景 4：SAP 事务（有领域知识加速）

```
用户：对 PO <PO_NUMBER> 执行收货

AI 执行过程（利用 SAP 领域知识）：
1. 输入事务码 /nMIGO → Enter
2. 填 PO 号（用 click+pressSequentially，因为 SAP 字段 readonly）
3. 确认行项目 → 过账
4. 读状态栏凭证号

输出："收货成功，物料凭证 <MATERIAL_DOCUMENT_NUMBER>。"
```

注意：前 3 个场景完全不需要领域知识，AI 用通用策略直接搞定。SAP 场景因为有特殊的 readonly 机制，领域知识能避免一次试错。

## 七、技术选型与理由

| 技术 | 选型理由 |
|------|----------|
| **Playwright** | a11y tree 原生支持、多浏览器、iframe 处理强、CDP 连接 |
| **Claude 多模态** | 视觉理解力强、结构化输出好、中文支持 |
| **TypeScript** | 类型安全、async/await、Playwright 原生语言 |
| **Skill（Markdown）** | 零部署、AI 直接阅读、修改即生效 |
| **YAML Flow** | 可复用流程、非开发人员可配置 |

### 为什么不用 Selenium/Puppeteer？

Playwright 的 `page.accessibility.snapshot()` 是杀手锏——直接拿到结构化的可交互元素列表，不需要自己解析 DOM。这对 AI 理解页面至关重要。

### 为什么不用 GPT-4V？

Claude 的结构化输出更稳定（JSON 格式），多模态理解在 UI 场景下表现好，且支持更长上下文。技术上框架是模型无关的，换 GPT-4V 也能跑。

## 十、总结

本框架的核心价值：

1. **通用性** — 一套框架操作任意 Web 页面，不绑定系统
2. **零选择器** — 不依赖 CSS/XPath，不怕页面结构变化
3. **自主决策** — AI 实时看屏幕判断，处理意外情况
4. **低成本** — 一个 Markdown 文件 + API 费用，无需平台
5. **可扩展** — 新系统零代码适配，复杂系统加领域知识即可
