# AI + Playwright 通用企业 Web 自动化框架系列发布总目录

> 系列定位：这不是一个 SAP 专用脚本项目，而是一套 **面向企业后台系统的通用 Web 自动化框架**。SAP WebGUI 是第一个高复杂度实践样例，用来验证 Flow Engine、Adapter、Playwright Runtime、Report / Trace 和 AI Diagnose 这些抽象是否站得住。

## 适合读者

- 正在做企业后台、ERP、SAP、SRM、OA、CRM 自动化的开发者。
- 写过 Selenium / Playwright 脚本，但被选择器漂移、iframe、弹窗和流程维护折磨过的人。
- 想把一次脚本沉淀成通用框架，而不是每换一个系统就重写一套的人。
- 对 AI Agent 操作网页感兴趣，但不想把不可逆业务操作完全交给模型的人。
- 想把自然语言、Skill、CLI、YAML Flow 和浏览器自动化串起来的工程实践读者。

## 框架定位

这个框架分成两层：

```text
通用框架内核
  - Flow Engine
  - Action Registry
  - Browser Runtime
  - Report / Trace
  - AI Diagnose
  - Skill / Natural Language Entry

领域适配层 Adapter
  - SAP Adapter
  - OA Adapter
  - CRM Adapter
  - SRM Adapter
  - E-commerce Admin Adapter
```

当前已经完整展开的是 `SAP Adapter`，SRM 则作为第二 Adapter 完成了第一轮 production skeleton 验证。原因不是 SAP 是唯一目标，而是 SAP WebGUI 足够复杂：iframe、动态 ID、readonly 输入、Tab 校验、事务状态记忆、系统消息和不可逆操作都集中在里面，适合作为框架的第一块压力测试样例。

## 项目状态说明

本文系列基于一个内部 PoC / 小规模验证项目整理。当前真实工程证据来自 SAP WebGUI 第一个复杂 Adapter，以及 SRM 第二 Adapter 的 V3 收口。

| 维度 | 当前状态 |
|---|---|
| 主线 Flow | 14 个，`validate-flows` 当前 0 errors / 0 warnings |
| SAP ECC Adapter | 第一个复杂样例，用于验证 iframe、readonly、Tab 校验、消息栏和事务状态记忆 |
| SRM Adapter | query / create / confirm / generate / upload production skeleton 已完成第一轮收口 |
| 可信执行边界 | Flow Contract、Action Registry、Adapter Registry、Promotion Gate、Approval Gate 已落地 |
| 采集层 | Recording Pack / `compile-recording` V1 已落地，完整 Recorder 仍是后续增强 |
| Skill | `sap-ui-auto` 已更新到 v3.10，包含 SRM 能力边界和真实操作限制 |

已实现：Flow Engine、YAML 编排、Page Object、dry-run、trace、HTML report、batch、`screenshot_and_report`、Recording Pack、Flow / Adapter capability 扫描。

规划中：`ai_diagnose` 真实 Claude API 接入、OpenClaw Skill 部署、飞书自然语言触发。

## 一句话结论

企业 Web 自动化最稳的路线，不是给每个系统重写一套脚本，也不是把浏览器完全交给 AI，而是沉淀一套 **通用内核 + 领域 Adapter** 的框架：规则明确的流程自动执行，系统差异留给 Adapter，AI 只在需要理解、判断和兜底时介入。
