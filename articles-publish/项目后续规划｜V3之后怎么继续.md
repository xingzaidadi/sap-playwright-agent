## 1. 当前项目状态判断

### 1.1 一句话状态

```text
项目已经完成 V1~V3 的框架主线闭环。
当前不是“继续堆功能”的阶段，而是“发布成果、收口技术债、准备真实环境验证、再设计 V4”的阶段。
```

### 1.2 V1~V3 当前判断

| 阶段 | 目标 | 当前状态 | 结论 |
|---|---|---|---|
| V1 | 让自动化资产有来源，并能执行和出报告 | 已完成 | Recording Pack + Flow Engine + Evidence Report 闭环成立 |
| V2 | 让自动化从“能跑”升级为“可信执行” | 已完成 | Flow Contract、Action Registry、Adapter Registry、Promotion Gate、Approval Gate 成立 |
| V3 | 用第二 Adapter 验证框架是否通用 | 第一轮完成 | SRM query / create / confirm / generate / upload 已完成 production skeleton 验证 |

### 1.3 当前最准确的对外说法

可以说：

```text
这个项目已经完成了从 SAP 自动化脚本到通用企业 Web 自动化框架的 V1~V3 架构闭环，
并用 SRM 作为第二 Adapter 完成了第一轮生产骨架验证。
```

不应该说：

```text
SRM 全流程已经可以无人值守生产运行。
已经支持所有企业 Web 系统。
已经可以自动生成真实 SAP 发票。
```

原因：

```text
当前完成的是框架能力、静态验证、测试和 production skeleton。
真实业务执行验证仍需要测试环境、账号、业务单号、人工审批和回滚方案。
```

## 3. 当前不应该继续做什么

这一节很重要，后续恢复上下文时先看这里。

### 3.1 不要继续盲目加新系统

暂时不要立刻开：

```text
OA Adapter
CRM Adapter
供应商门户 Adapter
更多 SAP tcode
更多 SRM 业务动作
```

原因：

```text
V1~V3 刚完成主线闭环，现在继续加系统会稀释成果，
也会把“框架验证”重新拖回“堆脚本”。
```

### 3.2 不要执行真实不可逆 SRM 动作

禁止自动执行：

```text
create settlement
confirm settlement
generate invoice
reject settlement
confirm invoice
reject invoice
submit finance review
post / release / upload irreversible business data
```

除非同时满足：

- 明确测试环境；
- 明确测试账号；
- 明确测试单号；
- 明确人工审批；
- 明确回滚或作废方案；
- 明确用户当次授权。

### 3.3 不要把旧 Flow 直接升级成生产能力

旧 Flow 可以作为素材，但不能直接生产化。

特别是：

```text
flows/srm-invoice-confirm.yaml
flows/srm-invoice-reject.yaml
flows/srm-maintain-settlement.yaml
flows/srm-reject-settlement.yaml
```

处理原则：

```text
旧 Flow 只提供业务语义和踩坑证据；
真正进入主线必须重新 Recording Pack -> Automation Plan -> Promotion Gate。
```

### 3.4 不要提交文章目录

继续遵守：

```text
articles/
articles-publish/
```

默认不提交。

原因：

- 它们是本地发布素材；
- 不属于框架运行必需代码；
- 当前里面有大量过程文档，提交会污染工程主线。

### 3.5 不要使用 `git add .`

继续坚持：

```text
只显式 stage 目标文件。
```

原因：

当前工作区里长期存在：

- 本地文章；
- 旧 Flow 草稿；
- `.bak`；
- 实验脚本；
- 过程记录。

用 `git add .` 很容易把不该提交的东西带进去。

4. **Page Object 不是银弹：为什么 Adapter 才是系统边界**
   - 主案例：SAP ECC 和 SRM 的差异。
   - 核心观点：Page Object 管页面，Adapter 管系统能力。

5. **从 60 个 warning 到 35 个 warning：一次 Flow Scanner 驱动的架构收口**
   - 主案例：V3 scanner 数据。
   - 核心观点：静态扫描不是形式主义，它能推动旧脚本收口成业务能力。

## 7. Line C：真实环境验证线

### 7.1 真实验证前置条件

真实 SRM 验证不能由 agent 自己启动。

必须准备：

| 项 | 要求 |
|---|---|
| 环境 | 明确是测试 / UAT / 沙箱，不是生产 |
| 账号 | 有权限但可控 |
| 测试数据 | 可用于 query / create / confirm / generate |
| 审批 | 明确哪些步骤允许执行 |
| 回滚 | 明确如何作废或冲销测试数据 |
| 记录 | 每次执行保存 report、screenshot、raw evidence |
| 边界 | 不允许误触真实供应商、真实发票、真实财务过账 |

### 7.2 验证顺序

必须从低风险到高风险：

```text
Step 1：srmQuerySettlementStatus
Step 2：createSettlement dry-run / mock data review
Step 3：confirmSettlement test-only
Step 4：generateInvoice test-only
Step 5：组合流程人工编排验证
```

不要从 generate invoice 开始。

### 7.3 每次真实验证必须记录

每次验证都要记录：

```text
验证日期
环境
账号角色
测试单号
执行 Flow
输入参数
approval 记录
执行结果
截图
rawText evidence
失败原因
是否产生业务状态变更
是否需要回滚
```

建议未来新建：

```text
articles-publish/真实环境验证记录｜SRM.md
```

或者工程目录：

```text
validation/srm/
```

但工程目录是否提交，需要另行判断。

### 7.4 真实验证完成标准

不能只看“脚本没报错”。

完成标准应该是：

| 能力 | 完成标准 |
---|---|
| query | 能查询目标 settlement，证据包含 settlement number 和状态 |
| create | 能在测试环境创建 settlement，证据包含创建结果和 settlement number |
| confirm | 能在测试环境确认 settlement，证据包含确认状态或成功消息 |
| generate | 能生成测试 SAP 暂估发票，证据包含 invoice number 或成功消息 |
| approval | 每个不可逆动作都有人工审批记录 |
| report | 每次执行都有可复盘报告 |

## 9. Line E：V4 设计线

V4 现在不要马上开发，但可以开始设计。

### 9.1 V4 候选方向

候选方向有 5 个：

| 方向 | 价值 | 风险 | 建议 |
|---|---|---|---|
| 真实环境验证闭环 | 让 skeleton 进入真实可用 | 高 | 等测试环境准备好再做 |
| 录制采集增强 | 让 Recording Pack 更接近真实操作 | 中 | 值得做 |
| Eval / Regression | 防止 skill 和框架退化 | 中 | 值得做 |
| 多 Adapter 扩展 | 验证 OA / CRM | 中高 | 暂缓 |
| Legacy 退役 | 降低维护成本 | 中 | 分批做 |

### 9.2 我最推荐的 V4 方向

如果没有真实 SRM 测试环境，V4 不应该做“真实业务执行”。

更推荐：

```text
V4 = Recording Pack 采集增强 + Regression Eval + Legacy 治理
```

原因：

- 不依赖真实业务环境；
- 能继续提升框架质量；
- 能防止 agent 后续偷懒或回退；
- 能为真实环境验证做准备。

### 9.3 V4 可能的功能拆分

#### V4.1 Recording Pack 采集增强

目标：

```text
把录制从“手工写 SOP”增强为“结构化采集页面状态、操作意图、选择器候选、等待证据”。
```

候选功能：

- 自动生成 selector candidates；
- 保存 a11y snapshot；
- 保存每步截图；
- 保存 before / after DOM 摘要；
- 保存用户手工标注的业务意图；
- 保存等待条件；
- 生成更好的 action-notes；
- 生成风险提示。

不做：

```text
录制即回放。
```

继续坚持：

```text
Capture, not Playback.
```

#### V4.2 Regression Eval

目标：

```text
防止 Recording Pack、skill、Flow scanner、Promotion Gate 后续退化。
```

候选 eval：

- Recording Pack 编译是否生成完整 drafts；
- irreversible Flow 是否必须有 approval；
- draft 是否不能直接 promotion；
- SRM action 是否匹配 Adapter capability；
- legacy combined capability 是否不再被新 Flow 使用；
- skill 是否提醒不要执行真实不可逆动作；
- scanner warning 是否没有异常增加。

#### V4.3 Legacy 治理

目标：

```text
让历史脚本不再干扰新框架。
```

候选动作：

- 标注 `confirmAndGenerateInvoice` legacy；
- 归档旧 Flow；
- 删除无用 `.bak`；
- 把有价值的旧脚本转成文章案例；
- 把可复用逻辑重写成 Adapter/Page Object。
