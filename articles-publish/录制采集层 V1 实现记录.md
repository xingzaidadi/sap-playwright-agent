# 录制采集层 V1 实现记录：先做 Recording Pack，而不是完整 Recorder

> 一句话结论：录制能力不要从“自动回放”开始做，而要从“结构化采集包”开始做。先把人工 SOP、截图、selector 候选、等待证据和成功证据沉淀下来，再编译成 Flow / Action / Adapter / Page Object 草稿。

这次没有直接做完整的浏览器录制器。

原因很简单：SAP WebGUI 这种企业后台页面并不适合一上来就做“录制即回放”。它有 iframe、动态 ID、readonly 输入、服务端局部刷新、系统消息、弹窗和不可逆业务动作。如果先做回放，很容易得到一堆看起来能跑、但很难维护的脚本。

所以 V1 的目标被收窄成两个 CLI：

```bash
npm run record-flow -- query-po-history
npm run compile-recording -- recordings/query-po-history
```

第一个命令负责创建采集包。

第二个命令负责把采集包编译成草稿。

## V1 做了什么

新增模块：

```text
src/recording/recording-pack.ts
```

新增命令：

```text
sap-auto record-flow <flow-name>
sap-auto compile-recording <recording-dir>
```

新增 npm script：

```json
{
  "record-flow": "tsx src/cli.ts record-flow",
  "compile-recording": "tsx src/cli.ts compile-recording"
}
```

新增测试：

```text
tests/unit/recording-pack.test.ts
```

## record-flow 生成什么

示例：

```bash
npm run record-flow -- query-po-history
```

生成：

```text
recordings/query-po-history/
  recording.meta.json
  sop.md
  action-notes.md
  expected-result.md
  selector-candidates.json
  wait-evidence.json
  screenshots/
  a11y/
  drafts/
```

这些文件的职责是分开的：

| 文件 | 作用 |
|------|------|
| `recording.meta.json` | 描述流程名、系统、风险等级、是否需要人工确认 |
| `sop.md` | 记录业务 SOP，不写 selector |
| `action-notes.md` | 把人工动作翻译成候选业务动作 |
| `expected-result.md` | 记录成功证据和失败证据 |
| `selector-candidates.json` | 存 selector 候选，不直接进入 Flow |
| `wait-evidence.json` | 存等待条件和可观察证据 |
| `screenshots/` | 存页面截图 |
| `a11y/` | 存 accessibility tree 或结构化页面快照 |
| `drafts/` | 存编译出来的草稿 |

## compile-recording 生成什么

示例：

```bash
npm run compile-recording -- recordings/query-po-history
```

生成：

```text
drafts/flow.yaml
drafts/action-registry.md
drafts/adapter-method.ts
drafts/page-object-method.ts
drafts/review-checklist.md
```

注意，这些是草稿，不是生产代码。

V1 的关键不是“自动写对”，而是把一次人工流程拆成可 review 的自动化资产：

```text
业务流程 -> Flow
业务动作 -> Action Registry
系统差异 -> Adapter
页面细节 -> Page Object
证据和等待 -> Review Checklist
```

## 设计取舍

### 1. 不覆盖已有文件

`record-flow` 默认不会覆盖已经存在的文件。

这是为了避免用户已经补过 SOP、截图、selector 候选后，被重新生成的模板覆盖掉。

`compile-recording` 也默认不覆盖已有草稿，除非显式传：

```bash
npm run compile-recording -- recordings/query-po-history --force
```

### 2. Flow 里不放 selector

即使 Recording Pack 里采集了 selector，生成的 `flow.yaml` 草稿也不会把 selector 写进去。

Flow 只应该描述业务动作：

```yaml
steps:
  - id: query_po_history
    action: query_po_history
```

selector、iframe、键盘输入、等待策略都应该留在 Adapter / Page Object。

### 3. 不可逆动作要显式标记

`recording.meta.json` 里有两个字段：

```json
{
  "riskLevel": "read-only",
  "requiresHumanApproval": false
}
```

如果是发布、过账、删除、审批、结算、退货这类动作，应该改成：

```json
{
  "riskLevel": "irreversible",
  "requiresHumanApproval": true
}
```

这样后续编译器、执行器和 Skill 都能把它识别为不可自动跳过的关键节点。

## 这次踩到的小坑

### Windows 下 npm.ps1 被执行策略拦截

在 PowerShell 里直接跑：

```bash
npm run build
npx vitest run tests/unit/recording-pack.test.ts
```

可能会遇到：

```text
无法加载文件 npm.ps1，因为在此系统上禁止运行脚本
```

解决方式是用：

```bash
npm.cmd run build
npx.cmd vitest run tests/unit/recording-pack.test.ts
```

### 测试和构建会写 dist / cache

TypeScript 构建会写 `dist/`。

Vitest 会启动 worker，并写测试缓存。

在受限环境里，这类命令需要放开写权限，否则会报 `EPERM`，但这不是代码逻辑错误。

## V1 的边界

已经实现：

- 创建 Recording Pack。
- 编译 Flow / Action / Adapter / Page Object 草稿。
- 生成 review checklist。
- 默认不覆盖已有文件。
- 单元测试覆盖创建、编译和非法名称校验。

还没有实现：

- 浏览器事件监听。
- 自动录制点击和输入。
- 自动生成稳定 selector。
- 自动采集 Playwright trace。
- 自动采集 a11y tree。
- 自动判断业务语义。

这些应该放到 V2 / V3，而不是塞进 V1。

## 结论

这一步的价值不在于“录制器已经完整”，而在于框架有了自动化资产生产的入口。

过去的路径是：

```text
SOP / 图文 / 人工复盘 -> 手写脚本
```

现在的路径变成：

```text
SOP / 图文 / 人工录制 / trace
  -> Recording Pack
  -> Flow / Action / Adapter / Page Object 草稿
  -> 人工 review
  -> dry-run
  -> 真实执行
  -> report / trace / regression
```

这就是我认为录制能力在这个框架里的正确位置：

```text
录制是采集，不是回放。
```
