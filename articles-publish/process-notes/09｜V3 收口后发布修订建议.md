# 09｜V3 收口后发布修订建议

> 用途：给 `09｜从 V1 到 V3：我如何把 SAP 自动化项目收敛成通用企业 Web 自动化框架.md` 做发布前修订。
> 当前归档状态：历史发布修订记录，已归档到 `articles-publish/process-notes/`。其中关于 V3 收口、配图和发布顺序的建议，已部分吸收到 09 篇、00 总目录和 00-08 系列整理中。

## 1. 当前 09 篇最大的问题

09 篇原来的核心判断偏保守，仍然停留在：

```text
V3 准备启动。
```

但工程现在已经推进到：

```text
V3 第一轮收口完成。
```

所以文章需要更新定位，否则会低估当前项目进展。

## 2. 建议更新后的主线

推荐把 09 篇主线改成：

```text
V1：让自动化资产有来源
V2：让自动化执行有契约和边界
V3：用 SRM 作为第二 Adapter 验证通用框架
```

不要把 V3 写成“SRM 自动化已经完全上线”。

准确表述是：

```text
V3 已完成 SRM 第二 Adapter 的生产骨架验证；
真实 SRM 生产执行仍需要环境、账号、业务单号和人工审批。
```

## 3. 建议替换文章里的状态表

建议使用下面这张表：

| 阶段 | 目标 | 当前状态 | 关键证据 |
|---|---|---|---|
| V1 | 从 SOP / 图文 / 录制材料生成 Flow 草稿，并能执行和出报告 | 已完成 | Recording Pack、Flow Engine、HTML Report |
| V2 | 让框架具备企业级可信执行边界 | 已完成 | Flow Contract、Action Registry、Adapter Registry、Promotion Gate、Approval Gate |
| V3 | 用第二 Adapter 验证框架是否通用 | 第一轮收口完成 | SRM query / create / confirm / generate production skeleton，scanner 0 error |

## 4. 建议加入真实工程数据

可以直接写进正文：

```text
V3 收口过程中，Flow Scanner 的 warning 从 60 降到 35。
这不是因为我手动隐藏了 warning，而是因为三条旧 SRM 页面级 Flow 被业务级 Action + Adapter capability 替换掉了。
```

这句话可以支撑文章的可信度。

不要只写“架构更清晰了”，要写：

```text
旧页面级动作正在从 Flow 里消失。
```

## 5. 建议加入三个真实案例

### 案例 1：为什么先做 read-only

用：

```text
srm-query-settlement-status
```

要表达：

```text
第二 Adapter 不应该从不可逆动作开始验证。
先用 read-only 能力打通 Flow -> Action -> Adapter -> Page Object -> Evidence -> Scanner。
```

### 案例 2：为什么 confirm 必须单独拆出来

用：

```text
srm-confirm-settlement
```

要表达：

```text
确认 settlement 是业务状态变更，必须有独立 action 和 approval gate。
不能藏在 generate invoice 的组合脚本里。
```

### 案例 3：为什么 generateInvoice 不能顺手 confirm

用：

```text
srm-generate-invoice
```

要表达：

```text
生成 SAP 暂估发票应该只处理已确认 settlement。
如果它内部顺手确认 settlement，就重新退化成 confirmAndGenerateInvoice 这种大而全脚本。
```

## 6. 建议新增的金句

```text
旧脚本里最危险的不是 selector 不稳，而是业务边界不清。
```

解释：

```text
selector 不稳，通常只是执行失败；
业务边界不清，可能让确认、生成发票、提交审批被包装成一个看似简单的自动化动作。
企业后台自动化真正要防的是后者。
```

## 9. 必须避免的说法

不要写：

```text
已经完成 SRM 全自动化。
已经支持所有企业 Web 系统。
可以无人值守生成 SAP 发票。
旧 Flow 已全部清理完成。
```

应该写：

```text
已经完成 SRM 第二 Adapter 的第一轮生产骨架验证。
query / create / confirm / generate 已经有业务级 Flow、Action、Adapter 和 Page Object。
真实 SRM 执行仍需要人工审批和环境验证。
旧 Flow 还在，但已经被分级处理，不再作为新能力来源。
```

## 10. 发布前检查

发布前建议确认：

- 是否把 V3 状态改成“第一轮收口完成”
- 是否写清楚 `implemented != 已真实生产运行`
- 是否写入 scanner 数据：`60 -> 35`
- 是否强调 `requires_approval`
- 是否说明文章里的 SAP/SRM 是样例，通用框架才是主角
- 是否避免承诺“无人值守生产执行”
- 是否保留旧脚本作为踩坑素材，而不是吹成历史资产

## 11. 最终建议

09 篇应该作为整个系列的“阶段性成果文”，不是纯技术细节文。

它最该回答的问题是：

```text
我为什么说这个项目正在从 SAP 脚本变成通用企业 Web 自动化框架？
```

答案就是：

```text
因为同一套 Core 机制已经从 SAP ECC 迁移到了 SRM：
Recording Pack、Flow Contract、Action Registry、Adapter Registry、Promotion Gate、Approval Gate、Evidence、Scanner 都没有换。
换的只是 Adapter 和 Page Object。
```
