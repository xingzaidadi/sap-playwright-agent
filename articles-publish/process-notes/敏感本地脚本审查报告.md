本轮只做三件事：

1. 统计敏感模式命中数量，不记录具体值；
2. 按用途和风险分类；
3. 用精确路径加入 `.gitignore`，防止误提交。

没有执行真实 SAP/SRM 操作，也没有运行这些脚本。

## 扫描方法

本轮扫描只统计命中数量，不在报告中保留具体 URL、账号、业务单号、供应商编号、token 或 API key。

扫描类别：

| 类别 | 说明 |
|---|---|
| URL | `http://` / `https://` |
| Credential | `password` / `pwd` / `token` / `secret` / `api_key` / `Authorization` / `Bearer` |
| BusinessId | 真实感 PO、凭证号或 10 位以上业务编号 |
| Vendor | `vendor` / `supplier` / `供应商` 等供应商相关字段 |
| Email | 邮箱模式 |

## 文件级处置建议

### `src/query-po-type.ts`

定位：本地 PO 查询脚本。

处置：

- 不提交原文件；
- 如果后续需要保留查询思路，只能改写成脱敏 fixture 或文档示例；
- 不能保留真实 URL、PO、供应商或内部字段。

### `src/run-full-flow.ts.bak`

定位：旧 full flow 本地运行入口。

处置：

- 不提交原文件；
- 里面的 runner 思路已经被 Flow Engine、Flow YAML、report/trace 路线替代；
- 不再作为当前主线入口使用。

### `src/sap/operations/full-settlement-flow.ts.bak`

定位：旧端到端 procurement settlement 流程备份。

处置：

- 不提交原文件；
- 只作为历史业务链路参考；
- 有价值的部分只能拆成业务级 action / Adapter method / Page Object 后重新实现；
- 不允许把旧的大而全流程直接恢复为 production Flow。

### `src/sap/operations/playwright-atomic-workshop.ts`

定位：Playwright 原子操作实验脚本。

处置：

- 不提交原文件；
- 可以人工提炼其中通用的等待、输入、点击、截图、弹窗处理经验；
- 提炼时必须重新写成正式 helper，并补测试；
- 不能复制带 URL、业务编号或环境假设的片段。

### `src/sap/operations/srm-settlement-ops.ts.bak`

定位：旧 SRM settlement 页面操作备份。

处置：

- 不提交原文件；
- 其中 confirm / generate / upload / reject 等动作都涉及业务风险；
- 只能作为 V3/V4 legacy 退役和拆分评估的历史参考；
- 新主线必须继续使用拆分后的业务级 action，例如 `srm_confirm_settlement`、`srm_generate_invoice`、`srm_upload_po_scan`。

### `src/sap/operations/toolskit-data-api.ts.bak`

定位：外部数据 API 集成备份。

处置：

- 不提交原文件；
- 不把其中的 URL、表名、字段、凭据、供应商相关逻辑带入公开材料；
- 如果未来确实要接入数据 API，必须重新设计配置、凭据加载、日志脱敏、测试替身和错误处理。

## `.gitignore` 决策

本轮把 6 个文件用精确路径加入 `.gitignore`：

```text
src/query-po-type.ts
src/run-full-flow.ts.bak
src/sap/operations/full-settlement-flow.ts.bak
src/sap/operations/playwright-atomic-workshop.ts
src/sap/operations/srm-settlement-ops.ts.bak
src/sap/operations/toolskit-data-api.ts.bak
```

选择精确路径，而不是 `*.bak` 或 `src/**/*.ts` 这类宽泛规则，原因是：

- 避免误忽略未来有价值的工程文件；
- 明确这些文件是本轮审查过的本地敏感素材；
- 保留后续单独重写、脱敏、测试化的空间。
