# 后续选题池

这份选题池用于延展当前系列。主线仍然是：通用企业 Web 自动化框架，SAP 是第一个高复杂度实践样例。

## P0：最值得继续写

### 1. Recorder CLI：如何半自动采集人工操作

建议标题：

```text
从 Recording Pack 到 record-flow：通用 Web 自动化框架的录制 CLI 设计
```

核心问题：

```text
V1 已经有 Recording Pack 规范，下一步如何让工具自动保存截图、a11y tree、trace 和 step note？
```

可带走：

- `record-flow <name>` 命令设计
- step note 交互方式
- screenshot / a11y / trace 存储规范
- recording.json schema

### 2. Automation Compiler：如何从录制包生成 Flow + Adapter 草稿

建议标题：

```text
compile-recording：如何把录制包编译成 Flow、Action 和 Adapter 草稿
```

核心问题：

```text
Recording Pack 只是输入，如何自动生成可 review 的自动化草稿？
```

可带走：

- Flow draft 生成规则
- Action draft 生成规则
- Adapter draft 生成规则
- Page Object draft 生成规则
- Review checklist

### 3. Adapter Contract Test：如何测试一个系统 Adapter 是否合格

建议标题：

```text
别只测流程跑没跑通：系统 Adapter 应该有自己的契约测试
```

核心问题：

```text
SAP Adapter、OA Adapter、CRM Adapter 都是系统差异层，怎么证明它们没有污染 Core？
```

可带走：

- Adapter checklist
- contract test 模板
- mock page / dry-run 测试思路
- selector 泄漏检查
- Page Object God Object 检查

### 4. 从一次失败 trace 到 Adapter 修复：完整复盘

建议标题：

```text
一次 SAP 自动化失败复盘：我是怎么把异常沉淀进 Adapter 的
```

核心问题：

```text
失败不是终点，失败应该变成 Adapter 规则。
```

可带走：

- trace 复盘模板
- failure -> diagnosis -> adapter patch -> regression test 流程
- AI Diagnose 如何参与但不直接修复

### 5. Report / Trace：为什么自动化框架必须能复盘

建议标题：

```text
自动化只说“失败了”没有意义：Report + Trace 才是团队敢用的前提
```

核心问题：

```text
团队敢用自动化的前提，不是它永远不失败，而是失败后能复盘。
```

可带走：

- HTML report 字段设计
- step log 结构
- screenshot 命名规范
- trace 存储策略
- 失败归因字段

## 当前最优下一步

如果只选一篇继续写，建议写：

```text
Recorder CLI：如何半自动采集人工操作
```

原因：

- Recording Pack V1 已经落地，下一步最自然的是让采集半自动化。
- 它能把“文档规范”推进到“工具能力”。
- 它能真实补齐当前框架最大的下一阶段短板：自动化资产从哪里来。
- 它也能反哺后续 Automation Compiler，让录制素材变成 Flow / Action / Adapter 草稿。
