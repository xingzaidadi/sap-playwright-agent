## 2026-05-09：Action Registry V1

目标：把 FlowRunner 里的 action switch 拆出去，让 FlowRunner 只负责流程编排。

做了什么：

- 新增 `src/engine/actions/*`。
- 保留旧 YAML action 名，避免迁移已有 Flow。
- `FlowRunner` 从大 switch 改为 `actionRegistry.get(step.action).execute(...)`。
- 子 Flow 复用同一个 `ActionRegistry`。
- 新增 `tests/unit/action-registry.test.ts`。

- FlowRunner 不能继续膨胀，否则后面接 OA/CRM 时会变成第二个“巨型脚本入口”。
- action registry 先做最小闭环，不急着引入插件系统。

## 2026-05-09：SAP/SRM Adapter Interface

目标：修复 Adapter Registry V1 的边界漏洞：action 不应该拿到 Page Object。

问题：

- Adapter Registry V1 虽然把 `SAPBasePage` 从 FlowRunner 移走了，但 `sap-ecc` 返回的还是 `SAPBasePage`。
- `srm_operation` 仍然可以把 `SRMPage` 当 adapter 用。
- 这会导致 Page Object 重新泄漏到 action 层，后面接 OA/CRM 时很容易复制 SAP 的内部结构。

做了什么：

- 新增 `SapEccAdapter`。
- 新增 `SapSrmAdapter`。
- `sap-actions.ts` 只依赖 `SapEccAdapter`。
- `integration-actions.ts` 的 SRM action 只依赖 `SapSrmAdapter`。
- Page Object import 留在 adapter implementation 里。
- 新增 `tests/unit/adapter-boundary.test.ts` 防止 action 重新 import `src/sap/*`。
- skill 升级到 v1.8。

踩坑/设计判断：

- Page Object 是 Adapter 内部实现，不是框架边界。
- action 应该表达“调用某个系统能力”，不应该知道 iframe、selector、toolbar button 的实现细节。
- boundary test 很朴素，但能挡住最容易复发的架构退化。

## 2026-05-09：Flow Contract V1

目标：把 Flow YAML 的边界从“约定俗成”变成“可校验契约”。

问题：

- 旧 Flow 只有 `name / description / params / steps`。
- 系统域靠 action 名隐式判断，例如 `navigate_tcode` 默认就是 SAP。
- Flow 风险等级没有统一字段，只能从某些 step 的 `requires_approval` 反推。
- 录制生成 Flow 时，如果没有 contract，很容易把 selector、iframe、xpath 等页面细节直接塞进业务 Flow。

做了什么：

- 新增 `FlowMetadata`。
- 新增 `validateFlowContract()`。
- Flow metadata 增加 `schema_version`、`adapter/adapters`、`risk`、`boundary`。
- 风险等级定义为 `read_only`、`simulated_change`、`reversible_change`、`irreversible`。
- 对已跟踪的 SAP/SRM 样例 Flow 补充 metadata。
- 对不可逆 Flow 补齐 approval gate。
- 增加测试，校验已跟踪 Flow contract、不可逆 Flow 的 approval gate、selector 泄漏 warning。
- skill 升级到 v1.9。

踩坑/设计判断：

- Flow Contract 第一版不能直接阻断所有旧 Flow，否则会影响现有运行；所以没有 metadata 先给 warning，不作为硬错误。
- 不可逆 Flow 缺 approval gate 应该是 error，因为这会影响真实业务系统安全。
- selector 泄漏先做 warning，因为部分探索型 Flow 可能还需要临时保留页面细节；后续可以用 `boundary.allow_page_details` 显式标记。
- Contract 不应该替代 Action Registry / Adapter Registry，它只负责声明 Flow 的所有权、风险和边界。

## 2026-05-09：Recording Compiler 接入 Flow Contract

目标：让录制生成的 Flow 草稿天然带契约，而不是先生成一份无边界 YAML，再靠人工补 adapter/risk/approval。

问题：

- `compileRecordingPack()` 之前直接用字符串模板生成 `drafts/flow.yaml`。
- 生成的 Flow 没有 `metadata.schema_version`、`adapter`、`risk`。
- 草稿参数没有显式 `type`，不符合 Flow Contract V1 的结构要求。
- 录制阶段的 `riskLevel` 没有映射到 Flow 层的 `risk`。
- 编译结果没有留下 contract 校验证据，review checklist 只能靠人工记忆。

做了什么：

- `compileRecordingPack()` 先构造 `FlowDefinition` 对象，再序列化为 YAML。
- 录制 `riskLevel` 映射到 Flow Contract 风险等级：
  - `read-only` -> `read_only`
  - `write` -> `reversible_change`
  - `irreversible` -> `irreversible`
- 根据 recording meta 推导 adapter：
  - SAP WebGUI -> `sap-ecc`
  - SAP SRM / srm -> `sap-srm`
  - 其他 domain -> 标准化后的 domain 名称
- 对不可逆或显式需要人工审核的 recording 自动生成 `requires_approval` 和 `approval_reason`。
- 新增 `drafts/flow-contract.json`，保存 `validateFlowContract()` 的结果。
- review checklist 增加 contract 状态、warning 数、error 数。
- `recordings/README.md` 更新生成产物说明。
- skill 升级到 v2.0。

踩坑/设计判断：

- Recording Compiler 不应该输出“看起来能跑”的脚本，而应该输出“能被审查、能被 contract 检查”的草稿。
- 录制阶段的风险语言和 Flow Contract 的风险语言不能各说各话，需要明确映射。
- adapter 推导只能做保守默认值，真正进入生产前仍要人工 review。
- contract 校验结果要落盘，否则 review 过程没有证据链。

  - params type
  - result type
  - risk / approval reason / expected evidence

- Page Object 可以 type-only 引用 adapter params，但真实生产代码里仍应按最终模块边界调整文件位置。

## 2026-05-09：Draft Promotion Gate V1

目标：在“生成代码草稿”和“写入正式 Flow/Action/Adapter/Page Object 文件”之间增加一道晋升审查门，避免 AI 把草稿当成生产代码直接落库。

问题：

- Plan-to-Code Draft V1 已经能生成 typed Action / Adapter / Page Object 草稿，但这些草稿仍然只是 review 输入。
- 如果下一步让 AI 直接把草稿复制进生产目录，很容易遗漏 action 命名、adapter method、Page Object 边界、risk/approval、敏感信息等人工审查点。
- 生成稿的目标文件位置之前只存在于人的判断里，没有落成机器可读的 promotion 证据。

做了什么：

- 新增 `PromotionGate` / `PromotionGateCheck` 类型。
- Recording Compiler 新增 `drafts/promotion-gate.json`。
- Recording Compiler 新增 `drafts/promotion-checklist.md`。
- Promotion Gate 记录：
  - Flow / Action / Adapter / Adapter method
  - 目标生产文件路径
  - flow contract 是否有效
  - automation plan 是否有效
  - action name / adapter method / page object boundary / risk approval / evidence / sensitive data 是否需要人工 review
  - compiler 是否只生成草稿、没有写入生产文件
- Gate 状态规则：
  - 有 fail -> `blocked`
  - 有 warning 或 manual_review -> `ready_for_review`
  - 全部 pass -> `ready_for_promotion`
- 当前 compiler 生成的正常草稿默认进入 `ready_for_review`，不直接进入 `ready_for_promotion`。
- `recordings/README.md` 更新 Recording Pack 编译链路。
- `web-ui-auto` skill 升级到 v2.4。

踩坑/设计判断：

- Promotion Gate 不是第三套大框架，而是 Recording Pack 编译链路的“最后一道边界”。
- 这一步的价值不是自动写更多代码，而是明确“什么时候不能写生产代码”。
- `ready_for_review` 比 `ready_for_promotion` 更符合当前阶段，因为 selector、业务含义、审批风险、敏感数据仍然需要人审。
- SAP/SRM 的目标文件路径只能作为建议，真正 promotion 前仍然要按当前模块边界调整。
